# 小程序开发团队协作指南

## 🚀 快速开始

### 启动新项目
```bash
# 进入技能目录
cd ~/.openclaw/workspace/skills/miniapp-team

# 启动团队（示例：电商小程序）
python3 scripts/team-launcher.py "电商小程序" "电商" "商品展示,购物车,支付,订单管理" 6
```

### 与团队成员沟通

你可以直接与我（项目经理）沟通，我会协调整个团队。也可以单独与某个角色深入讨论：

**直接对团队说**：
- "我们要开发一个[类型]小程序，核心功能有..."
- "进入设计阶段，需要输出设计稿"
- "开始开发，前端先搭建框架"
- "测试阶段，QA编写测试用例"

**单独沟通**：
- "@PM 帮我分析这个需求"
- "@设计师 设计一个商品详情页"
- "@前端 实现登录功能"
- "@后端 设计用户表结构"
- "@QA 编写购物车测试用例"
- "@运维 配置生产环境"
- "@运营 制定上线推广方案"

## 📋 项目阶段

### 阶段一：需求与规划（第1周）
**负责人**：产品经理

你需要提供：
- [ ] 小程序类型（电商/社交/工具/O2O/...）
- [ ] 目标用户群体
- [ ] 核心功能列表
- [ ] 预算范围
- [ ] 期望上线时间

**输出**：
- PRD文档
- 用户故事
- 功能优先级
- 数据埋点方案

### 阶段二：设计（第2周）
**负责人**：UI/UX设计师

**输入**：PRD文档

**输出**：
- 信息架构图
- 交互流程图
- 高保真设计稿
- 设计规范
- 切图资源

### 阶段三：开发（第3-4周）
**负责人**：前后端开发

**并行工作**：
- 后端：数据库设计 + API开发
- 前端：页面开发 + 组件封装

**里程碑**：
- 第3周末：API接口完成
- 第4周中：页面开发完成
- 第4周末：联调通过

### 阶段四：测试（第5周）
**负责人**：测试工程师

**测试类型**：
- 功能测试
- 兼容性测试（多机型）
- 性能测试
- 安全测试

**质量标准**：
- 测试用例覆盖率 > 80%
- P0级Bug清零
- 性能指标达标

### 阶段五：上线（第6周）
**负责人**：运维工程师

**检查清单**：
- [ ] 生产环境部署
- [ ] 域名/SSL配置
- [ ] 监控告警配置
- [ ] 数据备份策略
- [ ] 提交小程序审核
- [ ] 上线验证

### 阶段六：运营（持续）
**负责人**：运营专员

**工作**：
- 数据分析
- 用户增长
- 内容运营
- 活动策划
- 反馈收集

## 👥 团队角色速查

| 角色 | 职责 | 常用工具 |

| 产品经理 | 需求、规划、PRD | Axure, Xmind |

## 📝 交付物清单

### 每个阶段必须产出

**需求阶段**：
- PRD文档（含用户故事、验收标准）
- 功能列表（含优先级）
- 数据埋点设计

**设计阶段**：
- 设计稿（含异常/空/加载状态）
- 切图资源（2x, 3x）
- 设计规范文档

**开发阶段**：
- 源代码（Git仓库）
- API文档
- 组件库

**测试阶段**：
- 测试用例集
- 测试报告
- Bug清单（含修复状态）

**上线阶段**：
- 上线检查清单
- 运维手册
- 回滚方案

## ⚠️ 风险预警

常见问题处理：

**需求变更**：
1. 评估影响范围
2. 调整排期
3. 同步所有角色

**技术阻塞**：
1. 技术预研先行
2. POC验证
3. 及时上报风险

**进度延期**：
1. 每日站会同步
2. 资源协调
3. 范围调整

**质量问题**：
1. 加强Code Review
2. 补充测试覆盖
3. 追溯根因

## 🎯 成功标准

项目成功的定义：
- ✅ 按时交付
- ✅ 质量达标
- ✅ 用户体验良好
- ✅ 数据指标符合预期
- ✅ 团队成长

## 📞 沟通规范

### 会议制度
- **每日站会**：15分钟，同步进度
- **周会**：1小时，回顾与计划
- **评审会**：按需，确保质量

### 文档管理
- 所有文档统一存放
- 版本控制（v1.0, v1.1...）
- 变更记录

### 沟通渠道
- 日常：即时通讯
- 正式：邮件确认
- 紧急：电话/语音

## 🚀 开始使用

告诉我你的小程序需求，我会立即启动团队！

例如：
> "我要做一个社区团购小程序，功能包括：商品展示、团购下单、团长管理、配送跟踪。预算10万，希望2个月上线。"

团队立即进入**阶段一：需求与规划**。
