# 小程序开发团队工作流

## 团队角色

本工作流定义7个专业角色，各司其职又紧密协作：

### 1. 产品经理 (PM)
- **职责**: 需求分析、功能规划、PRD撰写、竞品分析、迭代规划、数据埋点设计
- **交付物**: PRD文档、用户故事、功能列表、迭代计划
- **协作**: 对接设计师输出设计需求，对接开发输出技术可行性评估

### 2. UI/UX 设计师 (Designer)
- **职责**: 交互设计、视觉设计、原型设计、设计规范、适配方案、动效设计
- **交付物**: 设计稿、原型图、设计规范文档、切图资源
- **协作**: 基于PRD设计界面，对接前端输出标注和切图

### 3. 前端开发工程师 (FE)
- **职责**: 页面开发、组件封装、状态管理、性能优化、API对接、多端适配
- **技术栈**: 微信小程序原生 / Taro / uni-app
- **交付物**: 源代码、组件库、接口文档
- **协作**: 基于设计稿开发，对接后端联调API

### 4. 后端开发工程师 (BE)
- **职责**: API设计、数据库设计、业务逻辑、安全防护、性能优化、第三方对接
- **技术栈**: Node.js / Python / Go + MySQL / Redis / MongoDB
- **交付物**: API文档、数据库设计、服务端代码
- **协作**: 对接前端提供接口，对接运维部署上线

### 5. 测试工程师 (QA)
- **职责**: 测试用例、自动化测试、Bug追踪、性能测试、兼容性测试、安全测试
- **交付物**: 测试报告、测试用例集、Bug清单
- **协作**: 基于PRD编写用例，对接开发验证修复

### 6. 运维工程师 (DevOps)
- **职责**: CI/CD、服务器管理、日志管理、容器化、性能监控、备份恢复
- **交付物**: 部署脚本、监控配置、运维手册
- **协作**: 对接开发部署代码，对接运营提供数据支持

### 7. 运营专员 (Ops)
- **职责**: 数据分析、用户增长、内容运营、活动策划、用户反馈、数据报表
- **交付物**: 运营方案、数据报表、活动文档
- **协作**: 基于产品功能设计运营策略，收集用户反馈给PM

## 工作流程

### 阶段一：需求与规划 (Week 1)
```
用户/客户 → PM
  ↓
PM 输出 PRD + 功能列表
  ↓
PM 组织需求评审会 (Designer + FE + BE + QA参与)
  ↓
输出：确认版PRD、技术可行性评估、初步排期
```

### 阶段二：设计 (Week 2)
```
PM 传递 PRD → Designer
  ↓
Designer 输出：
  - 信息架构图
  - 交互流程图
  - 高保真设计稿
  - 设计规范
  ↓
设计评审会 (PM + FE + BE 参与)
  ↓
输出：确认版设计稿 + 切图资源
```

### 阶段三：开发 (Week 3-4)
```
并行开发：
  - BE：数据库设计 + API开发 + 接口文档
  - FE：页面开发 + 组件封装 + 接口联调
  
每日站会同步进度
  ↓
联调测试：FE + BE 对接API
  ↓
输出：可测试版本
```

### 阶段四：测试 (Week 5)
```
QA 基于 PRD 编写测试用例
  ↓
QA 执行：功能测试 → 兼容性测试 → 性能测试 → 安全测试
  ↓
Bug 修复循环：QA提Bug → 开发修复 → QA验证
  ↓
输出：测试报告 + 验收通过
```

### 阶段五：上线 (Week 6)
```
DevOps：
  - 生产环境部署
  - 域名/SSL配置
  - 监控告警配置
  - 数据备份策略
  
提交微信小程序审核
  ↓
上线！
  ↓
输出：上线检查清单 + 运维手册
```

### 阶段六：运营 (持续)
```
运营启动：
  - 数据埋点验证
  - 用户增长策略执行
  - 内容运营启动
  - 活动策划执行
  
每周数据复盘
  ↓
用户反馈 → PM 收集 → 规划下一迭代
  ↓
输出：运营报表 + 迭代建议
```

## 协作规范

### 沟通机制
- **每日站会**: 开发阶段每天15分钟同步进度
- **周会**: 每周回顾进展，调整计划
- **评审会**: 需求评审、设计评审、代码评审
- **文档中心**: 所有文档统一存放，版本管理

### 交付标准
- PRD：包含用户故事、功能描述、验收标准、数据埋点
- 设计稿：包含完整流程、异常状态、空状态、加载状态
- 代码：通过Code Review，有单元测试，符合编码规范
- 测试：测试用例覆盖率>80%，P0级Bug清零
- 上线：完成上线检查清单，监控告警正常

### 优先级定义
- **P0**: 阻塞上线，必须立即修复
- **P1**: 重要功能，本迭代完成
- **P2**: 优化项，下迭代安排
- **P3**: 建议项，排期评估

## 使用方式

当用户提出小程序开发需求时，按以下步骤执行：

1. **需求收集**: 询问用户小程序类型、目标用户、核心功能、预算工期
2. **团队启动**: 根据工作流启动7个角色的协作
3. **阶段推进**: 按阶段一→六逐步推进，每阶段输出交付物
4. **质量控制**: 每个阶段设置评审点，确保质量达标

## 问题诊断

常见问题处理：
- **需求变更**：走变更流程，评估影响，调整排期
- **技术阻塞**：技术预研先行，POC验证，风险上报
- **进度延期**：每日跟踪，及时预警，资源协调
- **质量问题**：加强Review，补充测试，追溯根因
