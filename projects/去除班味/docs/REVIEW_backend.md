# PRD v1.1 评审报告 - 后端开发工程师

> **评审日期**: 2026-04-30  
> **评审人**: 后端开发工程师  
> **评审对象**: PRD v1.1「去除班味」AI班味检测仪  
> **评审结论**: ⚠️ 有条件通过（需调整3项架构设计 + 补充2项安全策略）

---

## 一、技术风险点

### 🔴 风险1：AI服务调用成本与稳定性
**问题**: PRD规划了4个AI接口（图像识别、浓度评分、辣评生成、方案生成），单次用户检测可能触发3-4次AI调用。按预估成本¥0.1-0.5/次，5万DAU的成本压力巨大。
**建议**:
- 建立「AI调用熔断机制」：单用户每小时最多3次AI分析，超限走规则引擎
- 引入「结果缓存层」：相同图片特征（ perceptual hash）的检测结果缓存24小时
- 辣评生成优先走模板库（命中率目标80%），仅复杂场景走LLM
- 图像识别采用自研YOLO微调模型（一次性训练成本 vs 持续API调用）

### 🔴 风险2：帖子审核状态机的队列设计不足
**问题**: PRD设计了AI审核→人工复审的二级审核流程，但未定义审核队列的吞吐量、超时处理、审核员分配策略。
**建议**:
- 引入消息队列（RabbitMQ/Redis Stream）处理审核任务
- AI审核结果≤2秒（P99），人工复审≤4小时（SLA）
- 高风险内容（置信度>0.8）直接拒绝+记录，不入人工队列
- 中风险内容进入人工队列，按「时间优先+内容类型优先」排序

### 🔴 风险3：分享裂变追踪的数据一致性
**问题**: inviteCode链路涉及多步操作（点击→打开→注册→检测），任何一步失败都会导致数据不一致。且微信分享场景值(scene)传递存在丢失风险。
**建议**:
- 采用「事件溯源」模式记录完整分享链路
- inviteCode写入小程序本地缓存（wx.setStorageSync），作为fallback
- 建立对账机制：每日比对分享记录与实际转化数据
- 引入幂等设计：同一invitee只能被记录一次转化

### 🟡 风险4：图片存储成本与合规
**问题**: 用户图片默认不公开，但长期存储原始图片（5MB/张 × 5万DAU × 30天）约7.5TB/月，存储成本高且涉及隐私合规。
**建议**:
- 图片生命周期策略：原始图保留30天→自动转存缩略图→90天后删除
- 用户删除账号时级联删除所有图片（符合《个人信息保护法》）
- 敏感图片（含面部特征）采用服务端加密（SSE-S3）

### 🟡 风险5：检测任务状态机的持久化
**问题**: 检测任务涉及多步状态转换（上传→排队→分析→完成），如果服务重启或AI服务超时，任务状态可能丢失。
**建议**:
- 使用数据库+Redis双写保证状态一致性
- 引入分布式锁（Redis Redlock）防止重复分析
- 设置任务超时自动清理机制（24小时未完成标记为失败）

## 二、数据库优化建议

### 2.1 索引设计补充

| 表 | 当前索引 | 建议补充 | 理由 |
|----|---------|---------|------|
| users | - | `idx_invite_code`, `idx_level` | 邀请查询、排行榜 |
| detection_records | - | `idx_user_id_created_at`, `idx_score` | 历史查询、分数统计 |
| user_plans | - | `idx_user_id_status`, `idx_plan_id` | 我的方案、方案统计 |
| posts | - | `idx_status_sort_score`, `idx_user_id`, `idx_topic_tags` | 信息流、个人主页、话题页 |
| comments | - | `idx_post_id_created_at`, `idx_parent_id` | 评论列表、楼中楼 |
| notifications | - | `idx_user_id_is_read`, `idx_user_id_type` | 通知列表、未读数 |

### 2.2 分表/归档策略

**detection_records 表**:
- 预估数据量：5万DAU × 每人每周2次 = 约40万条/月
- 建议按 `user_id % 16` 分16张表，或按月份分区
- 3个月前数据归档到冷存储（ClickHouse/Elasticsearch）

**posts 表**:
- 日新增1000条，月增3万条
- 按创建时间月份分区
- 30天无互动的帖子软删除（is_deleted=1），90天后硬删除

### 2.3 JSON字段的规范化建议

当前PRD大量使用JSON字段存储结构化数据（elements、dimensions、achievements等），建议：

| JSON字段 | 建议 | 理由 |
|----------|------|------|
| users.profile | 保持JSON | 灵活扩展，查询频率低 |
| users.achievements | 拆分为独立表 | 需要按成就查询、统计 |
| detection_records.elements | 保持JSON | 结构多变，无需单元素查询 |
| posts.topic_tags | 拆分为关联表 | 需要按标签筛选、聚合 |

## 三、架构建议

### 3.1 微服务拆分

```
API Gateway (Kong/Nginx)
    ├── Auth Service      // 微信登录、Token管理
    ├── Detect Service    // 检测任务调度、状态机
    ├── AI Proxy Service  // AI服务代理、熔断、缓存
    ├── Plan Service      // 方案管理、打卡追踪
    ├── Social Service    // 帖子、评论、互动
    ├── Notify Service    // 通知推送、消息队列
    ├── Upload Service    // 文件上传、OSS代理
    └── Admin Service     // 内容审核、运营后台
```

### 3.2 缓存策略

| 数据 | 缓存层 | TTL | 策略 |
|------|--------|-----|------|
| 用户信息 | Redis Hash | 1小时 | 变更时失效 |
| 检测历史 | Redis ZSet | 30分钟 | 分页缓存 |
| 帖子详情 | Redis String | 10分钟 | 更新时失效 |
| 信息流 | Redis List | 5分钟 | 预生成 |
| 点赞状态 | Redis Set | 24小时 | 持久化异步 |
| AI分析结果 | Redis String | 24小时 | perceptual hash索引 |

### 3.3 消息队列应用

```
生产者                    队列                        消费者
─────────                ─────                       ────────
用户发布帖子    ──→    post_review_queue     ──→    AI审核服务
                        (topic: review)
                        
AI审核完成      ──→    notify_queue          ──→    通知推送服务
                        (topic: notify)
                        
用户打卡        ──→    achievement_queue     ──→    成就计算服务
                        (topic: achievement)
                        
邀请转化        ──→    invite_reward_queue   ──→    奖励发放服务
                        (topic: reward)
```

## 四、AI服务集成方案

### 4.1 调用链路

```
用户上传图片
    ↓
Upload Service → OSS
    ↓
Detect Service 创建任务
    ↓
AI Proxy Service 并发调用：
    ├── Vision API（图像识别）→ 返回元素列表
    ├── Scoring Engine（浓度评分）→ 返回分数+维度
    └── Text API（辣评生成）→ 返回文案
    ↓
Detect Service 聚合结果
    ↓
返回前端
```

### 4.2 降级策略

| 场景 | 降级方案 | 用户体验 |
|------|----------|----------|
| Vision API超时 | 使用规则引擎（基于图片EXIF/文件名特征） | 评分准确性下降20% |
| Text API超时 | 从模板库随机选择3条匹配文案 | 个性化程度下降 |
| 全AI服务不可用 | 纯规则引擎评分 + 固定模板辣评 | 基础功能可用 |

### 4.3 成本控制

| 措施 | 预期效果 |
|------|----------|
| 模板辣评命中率80% | 减少80% LLM调用 |
| 图片特征缓存24小时 | 减少40% Vision API调用 |
| 单用户日限3次分析 | 控制峰值成本 |
| 异步批处理 | 非紧急任务合并调用 |

## 五、安全与性能建议

### 5.1 安全策略

| 层面 | 措施 |
|------|------|
| 接口安全 | JWT Token + Refresh Token，Token过期2小时 |
| 防刷机制 | 单IP限流 100次/分钟，单用户操作限流 |
| 内容安全 | AI预审+敏感词库+用户举报三级防护 |
| 数据安全 | 图片AES加密存储，敏感字段脱敏 |
| 隐私合规 | 用户删除账号级联清理数据，提供数据导出 |

### 5.2 性能目标拆解

| 指标 | PRD目标 | 后端目标 | 方案 |
|------|---------|----------|------|
| 图片上传 | ≤3秒 | ≤2秒 | CDN边缘节点 + 分片上传 |
| AI分析 | ≤8秒 | ≤6秒 | 并发调用 + 缓存命中 |
| 页面首屏 | ≤1.5秒 | - | 前端负责，后端接口≤200ms |
| 信息流翻页 | ≤1秒 | ≤300ms | 缓存 + 预加载 |
| API响应P99 | ≤500ms | ≤300ms | Redis缓存 + 数据库优化 |
