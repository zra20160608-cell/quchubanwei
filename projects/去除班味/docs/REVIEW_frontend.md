## 一、技术风险点

### 🔴 风险1：AI分析8秒延迟的前端体验瓶颈
**问题**: PRD要求AI分析≤8秒，但微信小程序在请求超过5秒时极易触发超时或用户主动退出。当前PRD采用轮询方案（15秒/次，最多6次），最长可能等待90秒。
**建议**:
- 引入 **Server-Sent Events (SSE)** 或 **WebSocket** 推送分析进度，替代轮询
- 前端采用「伪进度」策略：根据历史平均耗时模拟进度条，最后1秒等待真实结果
- 支持「后台分析」模式：用户可离开小程序，通过服务通知推送结果

### 🔴 风险2：OSS直传的兼容性与安全性
**问题**: PRD设计前端直传OSS，但微信小程序对域名白名单、HTTPS、CORS有严格限制。且前端暴露OSS签名存在安全风险。
**建议**:
- 采用「后端代理上传」方案：前端→后端→OSS，牺牲少许性能换取安全
- 或采用「STS临时凭证」：后端生成带有限权限的临时Token，有效期5分钟
- 必须配置小程序 `uploadFile` 合法域名白名单

### 🟡 风险3：分享海报的客户端生成性能
**问题**: PRD要求海报生成≤2秒，但小程序中使用 canvas 生成复杂海报（含雷达图+多行文字+头像）容易超出2秒。
**建议**:
- 服务端生成海报：前端上传数据→服务端渲染→返回图片URL
- 或使用 `html2canvas` 方案在webview中渲染后截图
- 预加载海报模板资源，减少运行时IO

### 🟡 风险4：圈子信息流长列表性能
**问题**: 日新增帖子≥1000条，信息流需支持无限滚动。小程序中大量图片+视频的列表容易出现内存泄漏和卡顿。
**建议**:
- 采用「虚拟列表」方案，仅渲染视口内+缓冲区的帖子
- 图片懒加载 + 渐进式加载（先模糊占位图，再高清图）
- 列表项回收复用（recycle-view组件）

### 🟢 风险5：深色模式适配成本
**问题**: PRD在v1.3规划深色模式，但如果在v1.1不建立CSS变量体系，后期改造成本极高。
**建议**:
- v1.1就引入CSS变量（或Tailwind配置）管理色彩
- 所有硬编码颜色必须走设计Token

## 二、接口设计建议

### 2.1 建议统一接口响应格式
当前PRD中接口返回值描述不统一，建议强制采用：
```json
{
  "code": 0,
  "message": "success",
  "data": { ... },
  "requestId": "req_xxx"
}
```

### 2.2 分页接口规范
所有列表接口统一分页参数：
```
?page=1&limit=20&sort=hot&filter=topic:工位改造
```
统一返回：
```json
{
  "data": { "list": [...], "total": 100, "hasMore": true },
  "code": 0
}
```

### 2.3 AI分析接口建议改为异步推送
当前设计：前端轮询 `/api/detect/:id/status`
建议改为：
1. 前端POST创建任务，立即返回 `taskId`
2. 服务端通过 **微信订阅消息** 或 **SSE** 推送完成通知
3. 前端收到推送后拉取完整报告

### 2.4 缺少的接口
| 缺失接口 | 用途 | 优先级 |
|----------|------|--------|
| GET /api/user/stats | 个人统计数据（检测次数/平均分数/等级进度） | P1 |
| POST /api/posts/:id/report | 举报帖子 | P1 |
| GET /api/search | 帖子/话题搜索 | P2 |
| GET /api/trends | 班味趋势排行榜 | P2 |

## 三、前端架构建议

### 3.1 框架选型
**推荐**: Taro 3.x + React
**理由**:
- 支持微信小程序 + H5 + 支付宝小程序多端输出
- React生态成熟，TypeScript支持好
- 团队React技术栈上手快

**备选**: uni-app + Vue3（如果团队Vue经验更强）

### 3.2 状态管理
```
全局状态（Recoil/Jotai）:
├── userSlice        // 用户信息、等级、成就
├── detectSlice      // 当前检测任务状态
├── planSlice        // 方案执行状态、打卡记录
├── socialSlice      // 圈子信息流、帖子缓存
└── uiSlice          // 全局UI状态（toast、loading、主题）

本地状态（React useState）:
├── 页面级表单状态
├── 动画过渡状态
└── 临时UI状态
```

### 3.3 目录结构
```
src/
├── pages/           // 页面组件
│   ├── detect/      // 检测模块（4个页面）
│   ├── plans/       // 方案模块
│   ├── social/      // 社交模块
│   └── profile/     // 个人中心
├── components/      // 公共组件
│   ├── common/      // 按钮、卡片、弹窗
│   ├── charts/      // 雷达图、折线图
│   └── business/    // 辣评卡片、打卡按钮
├── hooks/           // 自定义hooks
├── services/        // API请求层
├── stores/          // 状态管理
├── utils/           // 工具函数
└── constants/       // 常量、枚举
```

## 四、性能优化建议

| 优化点 | 方案 | 预期效果 |
|--------|------|----------|
| 图片加载 | 七牛/阿里云OSS + CDN + WebP格式 + 渐进加载 | 首屏加载减少40% |
| 代码分包 | 按页面路由懒加载，vendor拆分为独立包 | 主包≤2MB |
| 数据缓存 | SWR/React Query缓存策略，检测历史本地存储 | 减少30%请求 |
| 分析预加载 | 用户选择图片后预上传+预分析（在确认页提前开始） | 减少2秒等待 |
| 列表虚拟化 | 只渲染视口内+缓冲区内容 | 内存减少60% |
