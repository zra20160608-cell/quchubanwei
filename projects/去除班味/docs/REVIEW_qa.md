## 一、测试风险点

### 🔴 风险1：AI分析结果不可预期，难以自动化断言
**问题**: AI图像识别和辣评生成结果具有随机性，同一图片多次分析可能得到不同元素识别和文案。传统断言方式（固定预期值）无法使用。
**建议**:
- 建立「结果范围验证」：分数在合理区间（0-100），元素数量≥0，辣评长度在10-100字符
- 引入「黄金数据集」：100张已人工标注的测试图片，验证AI识别准确率≥85%
- 辣评质量采用「规则校验」：不含敏感词、格式正确、非空

### 🔴 风险2：分享裂变链路跨会话验证困难
**问题**: 分享→点击→打开→注册→检测的完整链路涉及微信生态、多个用户会话，端到端测试难以在单一会话中完成。
**建议**:
- 拆分验证：分别测试「分享链接生成」「参数传递」「转化归因」三个子链路
- 使用微信开发者工具的「模拟分享」功能测试参数传递
- 后端建立「对账测试」：验证分享记录与转化记录的一致性

### 🟡 风险3：内容审核的边界情况复杂
**问题**: AI预审置信度阈值（0.1/0.5/0.8）需要大量边界测试。人工复审流程涉及审核员主观判断，难以标准化。
**建议**:
- 构建「审核测试集」：100条边界内容（正常/擦边/违规各30%，模糊10%）
- 定义审核员一致性指标：多人审核同一内容，一致率≥90%
- 建立审核回溯机制：被误判的内容可追踪审核记录

### 🟡 风险4：打卡状态机的并发测试
**问题**: 用户可能在多设备登录，或在弱网环境下重复点击打卡，导致状态异常。
**建议**:
- 设计并发测试：同一账号在两个设备同时打卡
- 设计弱网测试：断网后重连，验证数据一致性
- 引入幂等性测试：同一打卡请求重复提交3次，仅记录1次

### 🟢 风险5：微信小程序兼容性矩阵庞大
**问题**: 需兼容iOS/Android主流机型，且微信基础库版本差异大（2.19.x+）。
**建议**:
- 定义最小兼容矩阵：iPhone 12/小米13/华为Mate60 + 微信最新3个版本
- 核心功能（拍照/上传/分析）必须在所有兼容设备上通过

## 二、测试策略建议

### 2.1 分层测试策略

```
单元测试（Jest）
├── 业务逻辑：评分算法、匹配算法、排序算法
├── 工具函数：图片压缩、日期格式化、脱敏处理
└── 数据校验：接口入参校验、状态机转换

集成测试（Supertest）
├── API接口：24个接口的CRUD+权限验证
├── AI服务：Mock AI响应，验证降级逻辑
└── 数据库：事务一致性、级联删除

E2E测试（Miniprogram-Automator）
├── 核心流程：拍照→分析→分享（ happy path ）
├── 异常流程：上传失败、分析超时、网络断开
└── 社交流程：发帖→审核→点赞→评论

性能测试（JMeter/Artillery）
├── API响应时间：P50/P95/P99
├── 并发能力：100/500/1000并发用户
└── 内存泄漏：长时间运行稳定性
```

### 2.2 测试环境规划

| 环境 | 用途 | 数据 |
|------|------|------|
| 开发环境 | 开发自测 | 模拟数据 |
| 测试环境 | QA测试 | 脱敏生产数据 |
| 预发布 | 回归测试 | 生产镜像 |
| 生产环境 | 线上监控 | 真实数据 |

## 三、关键测试用例设计

### 用例1：拍照检测完整流程（Happy Path）
```
前置条件：用户已登录
步骤：
  1. 进入检测页
  2. 选择「工位拍照」场景
  3. 拍照上传（正常光线、清晰工位图）
  4. 确认图片，补充工作年限「1-3年」
  5. 等待分析完成
  6. 查看诊断报告
预期结果：
  - 上传耗时≤3秒
  - 分析耗时≤8秒
  - 报告包含：分数、等级、雷达图、≥1个元素、≥1条辣评
  - 分享海报可正常生成
```

### 用例2：AI分析超时降级
```
前置条件：Mock AI服务响应延迟20秒
步骤：
  1. 上传图片并触发分析
  2. 等待15秒
  3. 观察前端表现
预期结果：
  - 前端显示「分析时间较长」提示
  - 后台轮询6次后自动降级
  - 返回基础评分（非AI评分）
  - 展示「服务繁忙」友好提示
```

### 用例3：分享裂变追踪完整链路
```
前置条件：用户A已登录，用户B未注册
步骤：
  1. 用户A完成检测并分享（带inviteCode）
  2. 用户B点击分享链接
  3. 用户B完成注册
  4. 用户B完成首次检测
  5. 查看用户A的邀请统计
预期结果：
  - 分享链接携带正确参数
  - 用户B的首次检测归因到用户A
  - 用户A的inviteCount+1
  - 奖励正确发放
```

### 用例4：帖子审核流程
```
前置条件：用户已登录
步骤：
  1. 用户发布含擦边内容的帖子
  2. 帖子进入AI审核
  3. AI判定为中风险（置信度0.6）
  4. 帖子进入人工复审队列
  5. 审核员判定通过
  6. 帖子正式发布
预期结果：
  - AI审核耗时≤2秒
  - 中风险内容不直接发布
  - 人工审核后状态变为APPROVED
  - 用户收到审核通过通知
```

### 用例5：打卡状态机转换
```
前置条件：用户已选择方案
步骤：
  1. 第1天：执行打卡 → 状态 IN_PROGRESS
  2. 第2天：未打卡 → 状态 AT_RISK（提醒推送）
  3. 第3天：补卡 → 状态 RECOVERED
  4. 持续打卡至最后1天 → 状态 COMPLETED
  5. 提交复测 → 状态 RE_TEST_READY
预期结果：
  - 每个状态转换记录正确
  - 补卡次数限制3次
  - 中断2天触发AT_RISK
  - 最终生成对比报告
```

### 用例6：内容安全-违规拦截
```
前置条件：用户已登录
步骤：
  1. 用户发布含敏感政治词汇的帖子
  2. 触发AI预审
预期结果：
  - AI置信度≥0.8，直接拒绝
  - 用户收到「内容违规」提示
  - 帖子状态为REJECTED
  - 记录到违规日志
```

### 用例7：图片上传异常处理
```
前置条件：用户已登录
步骤：
  1. 选择>5MB的图片
  2. 选择非jpg/png格式的图片
  3. 上传过程中断开网络
  4. 上传过程中服务器返回500
预期结果：
  - >5MB图片：前端拦截，提示压缩或裁剪
  - 格式错误：前端拦截，提示仅支持jpg/png
  - 网络断开：自动重试2次，提示检查网络
  - 服务器错误：提示稍后重试
```

### 用例8：并发点赞测试
```
前置条件：帖子存在，100个并发用户
步骤：
  1. 100个用户同时点赞同一帖子
  2. 100个用户同时取消点赞
预期结果：
  - 最终likeCount与Redis Set基数一致
  - 无负数count
  - 每个用户只能点赞1次（幂等）
```

### 用例9：弱网环境核心功能
```
前置条件：网络限速3G（1Mbps/延迟300ms）
步骤：
  1. 拍照上传
  2. 查看圈子信息流
  3. 发布帖子
预期结果：
  - 上传可完成（耗时增加但不超过30秒）
  - 信息流优先展示文字，图片渐进加载
  - 发帖可完成，本地缓存待网络恢复后自动同步
```

### 用例10：用户等级升级
```
前置条件：用户Lv.1，exp=0
步骤：
  1. 完成1次检测 → exp+10
  2. 分享1次 → exp+5
  3. 发布1条帖子 → exp+15
  4. 获得1个点赞 → exp+2
  5. 重复以上操作直到exp≥100
预期结果：
  - 每次行为正确累加exp
  - exp≥100时自动升级至Lv.2
  - 升级触发成就通知
  - 新权益解锁（如详细雷达图）
```

### 用例11：数据埋点完整性
```
前置条件：埋点系统已接入
步骤：
  1. 执行完整用户旅程：进入→拍照→分析→分享→发帖→点赞
  2. 检查所有埋点事件上报
预期结果：
  - 核心事件100%上报（app_launch, photo_upload, analysis_complete, report_share, post_publish, post_interact）
  - 事件属性完整
  - 上报成功率≥99%
```

### 用例12：多端数据一致性
```
前置条件：用户在两台设备登录
步骤：
  1. 设备A：完成检测，查看报告
  2. 设备B：进入小程序，查看检测历史
  3. 设备A：选择方案并开始打卡
  4. 设备B：查看我的方案
预期结果：
  - 设备B实时同步设备A的操作结果
  - 检测历史、方案进度、打卡记录一致
  - 无数据冲突或丢失
```

## 五、验收标准补充建议

| 验收项 | PRD当前描述 | 建议修改为 |
|--------|------------|-----------|
| AI分析耗时 | ≤8秒 | P50≤5秒, P95≤8秒, P99≤12秒 |

## 六、对PRD的明确反馈

### ✅ 通过的章节
- 第8章 验收标准：框架完整，大部分可量化
- 第10章 异常处理：8种异常场景覆盖全面
- 第6章 数据埋点：事件覆盖核心流程

### ⚠️ 需完善的章节
- **第8章 验收标准**：部分指标需要补充分位数标准（P50/P95/P99）
- **第12章 状态机**：需要补充「异常状态转换」测试点（如从ANALYZING直接到ABANDONED）
- **第10.1.3章 异常处理**：需要明确每种异常的前端提示文案（作为验收依据）

### ❌ 缺失的内容
- 缺少「测试数据准备」方案（黄金数据集、Mock数据生成）
- 缺少「兼容性测试矩阵」定义
- 缺少「安全测试用例」（XSS、SQL注入、越权访问）
- 缺少「性能测试基准」定义

---

> **总体评价**: PRD v1.1在功能验收和异常场景上覆盖度较高，但测试维度缺少自动化策略、兼容性矩阵和安全测试方案。建议补充上述内容后，由QA团队制定详细测试计划。
